Sistema Cognitivo usando MCP y Bases de Datos

Diseño e implementación de un servidor MCP en Python con FastMCP y SQLite3, que expone operaciones CRUD y consultas analíticas sobre una base de datos de inventario como herramientas externas para modelos de lenguaje.

Curso: Computación Cognitiva para Big Data
Tema: Model Context Protocol
Stack: Python 3.10 · SQLite3 · FastMCP

1. Presentación

Las organizaciones modernas requieren sistemas capaces de consultar, procesar y analizar información almacenada en bases de datos, servicios web y plataformas externas. En este contexto, los sistemas cognitivos apoyados en modelos de lenguaje pueden extender sus capacidades mediante herramientas que les permitan interactuar con datos reales.

El Model Context Protocol (MCP) permite conectar modelos de lenguaje con herramientas externas, tales como bases de datos, archivos, APIs y servicios computacionales. En esta actividad se desarrolló un servidor MCP que permite gestionar información de un inventario mediante operaciones CRUD y consultas analíticas básicas.

¿Qué es MCP? Es un protocolo abierto creado por Anthropic que estandariza la comunicación entre un modelo de lenguaje grande (LLM) y herramientas externas. El modelo puede "llamar" funciones en tiempo real y recibir sus resultados como parte de su razonamiento.

2. Objetivo de la actividad

Diseñar e implementar un servidor MCP en Python que permita realizar operaciones CRUD y consultas analíticas sobre una base de datos de inventario, con el propósito de comprender cómo un sistema cognitivo puede interactuar con herramientas externas.

5. Implementación

5.1 database.py — Creación de la base de datos

Crea la tabla productos con los campos requeridos usando CREATE TABLE IF NOT EXISTS, lo que permite reiniciar el servidor sin perder datos.

🗄️ database.py
import sqlite3

DB_NAME = "inventory.db"

def init_db():
    conn = sqlite3.connect(DB_NAME)
    cursor = conn.cursor()

    cursor.execute("""
        CREATE TABLE IF NOT EXISTS productos (
            id        INTEGER PRIMARY KEY AUTOINCREMENT,
            nombre    TEXT    NOT NULL,
            categoria TEXT,
            cantidad  INTEGER NOT NULL,
            precio    REAL    NOT NULL
        )
    """)

    conn.commit()
    conn.close()

5.2 server.py — Operaciones CRUD

⚙️ server.py — configuración inicial
from mcp.server.fastmcp import FastMCP
import sqlite3
from database import init_db, DB_NAME

init_db()
mcp = FastMCP("InventarioDB")

def get_connection():
    return sqlite3.connect(DB_NAME)
⚙️ server.py — herramienta crear_producto
@mcp.tool()
def crear_producto(nombre: str, categoria: str,
                    cantidad: int, precio: float) -> str:
    conn = get_connection()
    cursor = conn.cursor()
    cursor.execute(
        "INSERT INTO productos (nombre, categoria, cantidad, precio)"
        " VALUES (?, ?, ?, ?)",
        (nombre, categoria, cantidad, precio)
    )
    conn.commit()
    conn.close()
    return "Producto creado exitosamente"
⚙️ server.py — herramienta consultar_producto
@mcp.tool()
def consultar_producto(id: int) -> dict:
    conn = get_connection()
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM productos WHERE id = ?", (id,))
    row = cursor.fetchone()
    conn.close()
    if row:
        return {"id": row[0], "nombre": row[1],
                "categoria": row[2], "cantidad": row[3], "precio": row[4]}
    return {"error": "Producto no encontrado"}
⚙️ server.py — actualizar, eliminar y listar
@mcp.tool()
def actualizar_producto(id: int, cantidad: int) -> str:
    conn = get_connection()
    cursor = conn.cursor()
    cursor.execute("UPDATE productos SET cantidad = ? WHERE id = ?",
                   (cantidad, id))
    conn.commit(); conn.close()
    return "Producto actualizado correctamente"

@mcp.tool()
def eliminar_producto(id: int) -> str:
    conn = get_connection()
    cursor = conn.cursor()
    cursor.execute("DELETE FROM productos WHERE id = ?", (id,))
    conn.commit(); conn.close()
    return "Producto eliminado correctamente"

@mcp.tool()
def listar_productos() -> list:
    conn = get_connection()
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM productos")
    rows = cursor.fetchall()
    conn.close()
    return [{"id": r[0], "nombre": r[1], "categoria": r[2],
             "cantidad": r[3], "precio": r[4]} for r in rows]

5.3 server.py — Herramientas cognitivas y analíticas

⚙️ server.py — herramientas analíticas
@mcp.tool()
def calcular_valor_total_inventario() -> dict:
    conn = get_connection(); cursor = conn.cursor()
    cursor.execute("SELECT SUM(cantidad * precio) FROM productos")
    total = cursor.fetchone()[0]
    conn.close()
    return {"valor_total_inventario": total if total else 0}

@mcp.tool()
def productos_agotados() -> list:
    conn = get_connection(); cursor = conn.cursor()
    cursor.execute("SELECT * FROM productos WHERE cantidad = 0")
    rows = cursor.fetchall()
    conn.close()
    return [{"id": r[0], "nombre": r[1], "categoria": r[2],
             "cantidad": r[3], "precio": r[4]} for r in rows]

@mcp.tool()
def producto_mas_costoso() -> dict:
    conn = get_connection(); cursor = conn.cursor()
    cursor.execute("SELECT * FROM productos ORDER BY precio DESC LIMIT 1")
    row = cursor.fetchone()
    conn.close()
    if row:
        return {"id": row[0], "nombre": row[1], "categoria": row[2],
                "cantidad": row[3], "precio": row[4]}
    return {"error": "No hay productos registrados"}

@mcp.tool()
def estadisticas_inventario() -> dict:
    conn = get_connection(); cursor = conn.cursor()
    cursor.execute("SELECT COUNT(*), AVG(cantidad), AVG(precio),"
                   " SUM(cantidad * precio) FROM productos")
    tp, pa, pp, vt = cursor.fetchone()
    conn.close()
    return {"total_productos": tp, "promedio_cantidad": pa,
            "promedio_precio": pp, "valor_total": vt if vt else 0}

if __name__ == "__main__":
    mcp.serve()

7. Preguntas de análisis

1
¿Qué papel cumple MCP en la construcción de sistemas cognitivos?
El Model Context Protocol actúa como puente estandarizado entre un modelo de lenguaje y el mundo externo. Sin MCP, un LLM solo puede procesar texto dentro de su ventana de contexto estática. Con MCP, el modelo puede invocar herramientas (funciones Python, APIs, consultas SQL) de forma estructurada, recibir sus resultados e integrarlos en su razonamiento. Esto transforma al LLM de un generador de texto en un agente cognitivo capaz de actuar sobre datos reales y dinámicos. En esta actividad, el servidor MCP expone el inventario como un conjunto de capacidades que el modelo puede explotar sin conocer los detalles de implementación subyacentes.
2
¿Por qué es importante conectar un modelo de lenguaje con herramientas externas?
Los modelos de lenguaje tienen conocimiento estático limitado a su fecha de corte de entrenamiento y no pueden acceder a datos empresariales propietarios. La conexión con herramientas externas les permite: (a) acceder a información actualizada en tiempo real, (b) realizar operaciones con precisión exacta que los LLM no garantizan por sí solos (cálculos financieros, consultas sobre millones de registros), (c) persistir y modificar información más allá de la conversación, y (d) integrarse con sistemas empresariales reales como ERP, CRM o plataformas de inventario. Sin esta conexión, la utilidad del LLM en entornos empresariales quedaría severamente limitada.
3
¿Qué ventajas ofrece el uso de una base de datos frente al manejo de archivos planos?
Las bases de datos relacionales ofrecen ventajas sustanciales: (a) Integridad de datos con restricciones de tipo y llaves primarias que los CSV o JSON no pueden garantizar; (b) SQL como lenguaje de consulta que permite filtros, agrupaciones, joins y agregaciones sin cargar todos los datos en memoria; (c) Transacciones ACID que aseguran consistencia ante fallos o accesos concurrentes; (d) Índices que reducen el tiempo de búsqueda de lineal a logarítmico en grandes volúmenes; y (e) Escalabilidad: SQLite puede reemplazarse por PostgreSQL o MySQL con cambios mínimos en el código. Los archivos planos se vuelven inmanejables y propensos a corrupción a medida que el inventario crece.
4
¿Cómo podría ampliarse esta solución para un escenario de Big Data?
Para escalar a Big Data se podrían incorporar: (a) Motor de BD distribuido como PostgreSQL + Citus o Apache Cassandra para manejar terabytes de datos de inventario en múltiples nodos; (b) Apache Kafka para capturar flujos en tiempo real de movimientos de inventario (entradas, salidas, ajustes); (c) Apache Spark para procesamiento analítico distribuido sobre el historial de transacciones y generación de reportes predictivos; (d) Data Lake en S3 o Google Cloud Storage para almacenamiento económico de datos históricos; y (e) nuevas herramientas MCP como predecir_demanda(), detectar_anomalias() o generar_reorden_automatico() que aprovechen modelos de ML entrenados sobre los datos históricos.
5
¿Qué riesgos se deben considerar al permitir que un modelo de IA consulte o modifique datos?
Los principales riesgos son: (a) Inyección de prompts maliciosos que manipulen al modelo para ejecutar operaciones no autorizadas como eliminar todos los productos o exfiltrar datos sensibles; (b) Falta de autorización granular: el modelo puede invocar cualquier herramienta disponible sin verificar si el usuario tiene permiso para esa operación específica; (c) Alucinaciones con efectos reales: el modelo podría insertar datos incorrectos (precios negativos, cantidades erróneas) que afecten operaciones reales; (d) Ausencia de auditoría: sin logs detallados es imposible rastrear qué herramienta invocó el modelo y con qué parámetros; y (e) Denegación de servicio por consultas mal construidas que saturen la base de datos. Mitigaciones recomendadas: validación estricta de parámetros en cada tool, autenticación OAuth 2.0, principio de mínimo privilegio y registro completo de auditoría.

8. Conclusiones

El desarrollo de esta actividad permitió comprender en la práctica cómo los sistemas cognitivos modernos amplían su capacidad de acción a través de herramientas externas estandarizadas:

  • MCP como habilitador cognitivo: El protocolo transforma a los LLM de sistemas de generación de texto en agentes capaces de interactuar con datos reales, superando las limitaciones del conocimiento estático de entrenamiento.
  • Separación de responsabilidades: Dividir la lógica en database.py (persistencia) y server.py (herramientas) facilita el mantenimiento, las pruebas y la futura escalabilidad del sistema.
  • SQLite como prototipo: Resulta ideal para desarrollo y pruebas, pero en producción debe reemplazarse por un motor más robusto con soporte de concurrencia y respaldo.
  • Validación como seguridad básica: La verificación de parámetros de entrada en cada herramienta MCP es la primera línea de defensa contra operaciones incorrectas o malintencionadas.
  • Patrón replicable: La arquitectura LLM + MCP + BD es aplicable a inventarios, CRM, análisis financiero, soporte técnico y cualquier dominio empresarial que requiera razonamiento sobre datos estructurados.
